# ClimateTrust AI — Ethics & Responsible AI

## Our Commitment

ClimateTrust AI is built on the principle that **AI systems making judgements about data quality must be transparent, explainable, and fair**. Environmental data affects policy, funding, and scientific credibility — getting it wrong has real consequences.

This document outlines our ethical commitments, known limitations, and mitigation strategies.

---

## Core Ethical Principles

### 1. No Black-Box Rejection

Every anomaly flag includes a machine-readable reason code AND a human-readable explanation. We never simply discard or reject data without explaining why. The pipeline outputs:

```json
{
  "alerts": [
    {
      "timestamp": "2026-06-03T14:00:00",
      "reasons": [
        "Temperature spike (z=4.2)",
        "Drift from reference (+3.1°C)"
      ]
    }
  ]
}
```

Users always know exactly what triggered a flag.

### 2. Correction Over Suppression

The platform **calibrates and improves data**, not hides it. Even a sensor with a Trust Score of 20/100 (Grade F) has its corrected readings published alongside the original — giving researchers and end-users both views.

### 3. Transparent Scoring

The Trust Score formula is fully documented, deterministic, and auditable. There are no hidden weights, proprietary models, or unexplainable ML layers in the scoring itself. The formula is printed in `README.md`, `ARCHITECTURE.md`, and in the dashboard UI.

### 4. Uncertainty Disclosure

The system explicitly discloses when its outputs are less reliable:

- If `reference_temp_c` data is unavailable (Open-Meteo fetch failed or no overlap with sensor timestamps), calibration is marked `applied: false` and the trust score is lower-bounded conservatively.
- The diagnostic explanation includes the `source` field (`"llm"` vs `"template"`) so users know whether the narrative was dynamically generated or deterministic.

### 5. No Training on User Data

Uploaded CSV files are processed in-memory and are never stored, logged, or used to retrain any model. The Isolation Forest model is re-fit from scratch on each request. There is no persistent storage of user sensor data.

## Known Limitations

| Limitation | Impact | Mitigation |
|------------|--------|------------|
| Open-Meteo provides ambient air temperature, not sensor-level microclimate | Reference may not match indoor or elevated sensors exactly | Clearly labelled as "reference" not "ground truth" |
| Isolation Forest contamination parameter fixed at 5% | May over-flag on very noisy datasets or under-flag on clean ones | Exposed in API response; contamination is a known scikit-learn parameter |
| Linear calibration assumes stationary bias | Will not correct non-linear sensor degradation | Drift detrend step partially addresses this; documented in calibration_meta |
| LLM explanations may hallucinate | Text-based models can generate plausible-sounding but inaccurate details | All quantitative claims in the prompt (not generation); template fallback always available |
| 168-row demo datasets are synthetic | Results on real sensor deployments may differ | `generate_sensors.py` uses realistic diurnal cycles; real data encouraged |
| No support for PM2.5, CO2, or non-temperature sensors yet | Limits applicability to multi-pollutant air quality networks | Roadmap item; humidity is supported today |

## Bias & Fairness Considerations

### Geographic Bias
The default demo location is **Delhi, India (28.6°N, 77.2°E)** — chosen because it is directly cited in the MSV-2035 Instrument Crisis context. Open-Meteo supports any global coordinates; users can specify their own latitude/longitude.

### Socioeconomic Bias
Low-cost sensors used by citizen scientists and NGOs are more likely to drift and spike than expensive research-grade instruments. This means community-contributed data may systematically receive lower trust scores.

**Our response:** The platform's purpose is precisely to help these sensors get calibrated and corrected — not to penalise them. The calibration engine brings low-cost sensor output closer to reference-grade, and the AI explanation explicitly names calibration as a recommended action rather than data rejection.

### Language Bias
Diagnostic explanations are currently generated in English only. This is a known limitation for South Asian research teams whose primary language may be Hindi, Tamil, Bengali, or another Indian language. Roadmap: multi-language support via LLM's native multilingual capabilities.

## Reporting Issues

If you identify a case where the trust scoring, anomaly detection, or diagnostic explanation produces a result that seems unfair, incorrect, or misleading — please open a GitHub issue with:

1. The input CSV (or a description of the sensor type)
2. The result you received
3. The result you expected

We are committed to iterating on these systems responsibly.
